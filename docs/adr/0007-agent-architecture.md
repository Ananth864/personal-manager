# Agent architecture: Vercel AI SDK, custom domain tools over the service layer, direct execution

The cooking agent is a single reactive assistant built on the Vercel AI SDK, running server-side in a TanStack Start route handler under your Clerk session, streaming to the chat UI via `useChat`. It uses OpenAI's `gpt-5.6-luna` through the SDK's provider abstraction, so the model is a config swap, not a rewrite. The model never touches the database directly; it calls a fixed set of 12 domain-operation tools (`assign_slot`, `clear_slot`, `add_recipe`, `add_ingredient`, `restock_ingredient`, `set_ingredient_state`, `query_schedule`, `query_inventory`, `query_recipes`, `query_ingredients`, `query_food_bank`, `query_past_weeks`) that run the *same* service-layer logic the UI uses. The tool list is the capability boundary — notably absent are `trigger_cook`, recipe edit/delete, and any DB/SQL access, so the autonomy rules from ADR 0003 are enforced structurally, not by prompt.

Each turn, a fresh state snapshot (current Week's schedule + Inventory states/quantities + Food Bank portions) is injected alongside the last ~5 turns of conversation history from the `cooking_chat_messages` table; tool *results* remain authoritative as the model acts within a turn. Writes execute directly on instruction (no proposal gate) — safety comes from tool validation, streaming visibility, the free mutability of the Schedule, and `created_by` audit. Each tool call is atomic; multi-call sequences (e.g. "plan my week" → 12 `assign_slot` calls) are not transactional in v1 — partial failures surface to the model for retry or to the user for correction.

Rejected alternatives: raw DB/SQL access for the agent (unsafe — the model will eventually produce a damaging write); generic CRUD pass-through (multi-step ops become fragile compositions); LangChain/Mastra (abstraction layers without value on this stack); propose-and-approve for every action (friction on instruction-driven requests); multi-step DB transactions (complexity premature for bounded personal-app operations); single-vendor model lock-in (rejected via the SDK abstraction).
